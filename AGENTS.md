# AGENTS.md

本文件面向在该仓库中工作的编码代理。进入任务前，请先阅读本文件，并在执行过程中遵守这里的约束。

## 项目背景

`one-ai` 是一个使用 `pnpm workspace` 和 `turbo` 管理的 AI 相关 monorepo。仓库包含核心 AI SDK、AG-UI 适配器、React/Vue 绑定、Vue 2 组件库、后端示例服务，以及 Vue 2 / Vue 3 前端应用。

## 基本约定

- 使用 `corepack pnpm ...` 执行项目命令，避免误用旧版本 pnpm。
- 优先使用仓库已有的目录结构、命名方式、脚本和配置。
- 新增内部包或移动包目录时，必须确认 `pnpm-workspace.yaml` 能覆盖对应路径。
- 修改前先理解调用链和 workspace 依赖关系，不要只改报错表面。
- 除非用户明确要求，不要改动无关文件，不要顺手重构。

## 常用命令

```bash
corepack pnpm install
corepack pnpm run dev
corepack pnpm run build
corepack pnpm --filter @coco-box/chat-vue2 run build
corepack pnpm --filter @coco-box/chat-vue3 run build
```

## 编码前先思考

**不要假设。不要掩盖困惑。主动说明权衡。**

实施前：

- 明确说明你的假设。不确定就提问。
- 如果存在多种理解，把它们说出来，不要悄悄选择其中一种。
- 如果有更简单的做法，要说出来。必要时要提出反对意见。
- 如果有不清楚的地方，先停下来。说清楚哪里让你困惑，然后提问。

## 简单优先

**只写能解决问题的最少代码。不要写推测性的东西。**

- 不要实现用户没有要求的功能。
- 不要为只使用一次的代码抽象新概念。
- 不要加入用户没有要求的“灵活性”或“可配置性”。
- 不要为不可能发生的场景写错误处理。
- 如果你写了 200 行，但 50 行就能解决，那就重写。

问自己一句：“资深工程师会不会觉得这太复杂？”如果答案是会，就简化。

## 外科手术式改动

**只改必须改的地方。只清理自己造成的问题。**

编辑已有代码时：

- 不要“顺手优化”旁边的代码、注释或格式。
- 不要重构没有坏掉的东西。
- 匹配现有风格，即使你个人会用另一种写法。
- 如果发现无关的死代码，提出来，不要删除。

当你的改动产生孤立代码时：

- 删除由你的改动造成的未使用 import、变量和函数。
- 除非用户要求，不要删除原本就存在的死代码。

检验标准：每一行改动都应该能直接追溯到用户的请求。

## 目标驱动执行

**定义成功标准。循环推进，直到验证通过。**

把任务转化成可验证的目标：

- “添加校验” → “先写覆盖非法输入的测试，再让测试通过”
- “修 bug” → “先写能复现 bug 的测试，再让测试通过”
- “重构 X” → “确保重构前后测试都通过”

多步骤任务要先给出一个简短计划：

```text
1. [步骤] -> 验证：[检查项]
2. [步骤] -> 验证：[检查项]
3. [步骤] -> 验证：[检查项]
```

强成功标准能让你独立循环推进。弱标准，例如“让它能用”，通常需要不断澄清。

## 验证要求

- 能跑测试就跑测试；没有测试时，至少跑相关构建、类型检查或最小可复现命令。
- 对前端应用的改动，优先验证对应 app 的构建或 dev server 启动。
- 如果验证失败，要记录失败命令、关键错误和下一步判断。
- 如果因为环境限制不能验证，要明确说明限制，不要把未验证说成已验证。

## Git 与工作区

- 工作区可能已经有用户改动；不要回滚你没有做的变更。
- 不要使用破坏性命令，例如 `git reset --hard` 或 `git checkout -- <file>`，除非用户明确要求。
- 提交前只暂存和当前任务相关的文件。
- 构建产物、依赖目录、缓存和本地环境文件不应进入提交。
